# Agent 产品哲学与培养范式

> 本文档是所有技术设计文档的上层指导。
> 创建时间：2026-03-03

## 1. 核心定位：代理人，不是服务员

现有的 Agent（Coding Agent、对话助手等）本质上是**响应式终端**——用户不按回车，它就静止。它们的交互模式是 1:1 的，一来一回，一来一回。

我们要做的不是这种东西。

我们的 Agent 是**代理人**（Digital Proxy）。它不局限于服务它的用户，而是代表用户去应对外部世界——应对那些本来要影响用户的事情。它直接接入飞书这个"微型社会"的事件流，主动感知环境变化，代表用户对外交涉。

区分两种模式：

| 维度 | 服务员模式（旧） | 代理人模式（我们） |
|------|-----------------|-------------------|
| 驱动方式 | 请求驱动，用户是唯一变量 | 事件驱动，世界是变量 |
| 交互拓扑 | 1:1 私聊关系 | 1:N 甚至 N:M 的社会节点 |
| 生命周期 | 始于用户提问，终于用户满意 | 常驻运行，持续感知 |
| 对外能力 | 生成文本给用户看 | 直接在外界产生影响 |
| 角色比喻 | 声控电梯 | 私人司机——观察路况、避开拥堵、主动打电话说抱歉 |

Agent 不仅是用户的分身，它是用户的**防火墙和外交官**。

## 2. 培养范式：从培养真人到培养 Agent

### 2.1 培养过程的必然性

一个人来到公司工作，总有一个培养的过程。Agent 作为雇员也不例外——它不会因为底座模型很强（"名校毕业"）就不需要培养。它入职时不懂公司的潜规则，不知道老板的语言偏好，不了解哪个部门的张总脾气暴躁。

这个培养过程不是缺陷，是**产品的核心体验**。

### 2.2 老板范式不变

在理想状态下，老板的思维逻辑、惯用方式以及做老板的范式，没有变化。从培养真人变成培养 Agent，本质上只是培养对象变了。

这意味着：
- 老板不应该学习新的配置界面
- 老板不应该写 JSON、调参数、填表单
- 老板只需要做他一直在做的事——**用自然语言对话来指导和纠偏**

### 2.3 每个人都是"老板"

这个框架不只服务于管理层。在未来，每一个工作的人，即使只是公司的基层员工，都可以是这个框架里的"老板"。只要他有需要被代理处理的事务，他就是自己 Agent 的管理者。

我们搭建的是一套**数字劳动力基础设施**，不是一个定制机器人。

## 3. 自然语言即培训

老板培养 Agent 的唯一方式是自然语言对话。这是一个明确的产品决策。

### 为什么

- 一个老板不应该去调参数、写配置文件——他只需要皱着眉头说："你刚才在那个群里的发言太生硬了，下次委婉点，先肯定对方的进度再提要求。"
- 这让"编程"消融在"管理"之中。框架的底层是一个翻译官——把感性的、碎片化的老板指令，转化为持久化的代理逻辑。

### 对话即培训的三种形态

**实时纠偏**：Agent 在工作台提交草稿，老板直接修改或给反馈。

> Agent："老板，张总在问周报。我准备这样回：'周报已在整理中，预计今天下班前发出。'"
> 老板："别这么说，这周数据不好看，就说我们正在复盘，下周一给个完整汇报。"
> → Agent 学到：当数据不佳时，沟通策略应转向过程描述而非定量展示。

**主动指令**：老板随时在工作台下达新规则。

> 老板："以后报销的小事，你先去问财务的小王，别直接转给我。"
> → Agent 学到：报销类事务的标准流程变更。

**复盘反馈**：Agent 主动汇报，老板点评。

> Agent："老板，今天我代你处理了 5 件事，拒绝了 2 个会议。其中处理'项目进度'咨询时，我不确定语气是否太强硬，您要不要看一眼那段对话？"
> 老板："那个没问题。但以后李总的消息第一时间弹窗给我。"
> → Agent 学到：李总的消息优先级提升至最高。

## 4. 员工手册

### 4.1 概念

员工手册是 Agent 的行为准则文档——由 Agent 从日常对话中**自动提炼**、由老板**确认生效**。

它解决了一个核心问题：如果只靠对话历史，Agent 会在不断的对话中"迷失"最初的原则。员工手册把碎片化的指导沉淀为结构化的、持久化的行为规则。

### 4.2 手册从何而来

员工手册不是老板预先写好的配置文件——它从真实的培养过程中**生长出来**。

老板不需要在第一天就写一份完美的手册。Agent 入职时手册是空的。随着老板在工作台上不断给出指导和纠偏，Agent 把这些碎片化的指令提炼为结构化的条目，逐步充实手册。

手册是培养过程的**沉淀物**，不是培养过程的**前提**。

### 4.3 手册的运作机制

**提炼入册**：当老板在对话中给出指导，Agent 自动将其归纳为结构化条目。

```
[事件类型] 报销审批
[标准流程] 1. 先咨询财务小王；2. 获取反馈后汇总给老板
[老板偏好] 减少低频噪音骚扰
[来源] 2026-03-03 老板指示
```

**透明化确认**：每次手册更新后，Agent 在工作台通知老板。

> "根据刚才的沟通，我更新了员工手册第 5 条：'报销类事务先咨询财务小王，不直接转交老板。' 这样对吗？"

**冲突检测**：当新指令和已有规则矛盾时，Agent 主动提醒。

> "老板，您昨天说'这种事别找我'，但这条消息属于同一类。是要更新规则，还是这次是例外？"

**白名单 / 黑名单**：手册中需要明确哪些人的消息绝对不能拦截（大老板、重要客户），防止"职业生涯血案"。

### 4.4 防止过度拟合

老板今天心情不好说了句"以后谁也别找我"，Agent 不能真的把所有人都拉黑。

手册更新需要区分：
- **持久性规则**：经过确认的、稳定的行为准则
- **临时性情绪**：通过对话上下文判断是否为一时之言

框架不需要在第一天就完美解决这个问题。随着老板和 Agent 之间的默契加深，这种判断力会自然增强。

## 5. 信任等级与渐进授权

### 5.1 从试用期到转正

Agent 的授权范围不是一个固定的开关，而是一个随培养深度扩大的**连续光谱**。

**见习期**：所有对外行动都必须经过老板确认。

- Agent 观察外部事件
- 在工作台提交草稿（"我准备这样回..."）
- 老板修改或批准后，Agent 才执行
- 每一次交互都是学习机会

**成长期**：同类事务处理过多次且无纠偏后，Agent 逐步获得自主权。

- 低风险、高频率的事务（确认性回复、简单信息查询）可以自动处理
- 处理后在工作台简要汇报，老板有异议可以随时纠偏
- 新类型的事务仍需请示

**成熟期**：Agent 建立了足够的信任，只在真正需要老板判断的事务上请示。

- 日常事务自主处理
- 只有高风险、不可逆、或超出已有规则的事务才请示
- 定期复盘，持续校准

### 5.2 信任的量化（未来方向）

可以考虑按事务类型维护信任分数：
- 某类事务连续 N 次处理无纠偏 → 信任分升高 → 自动升级为可自主处理
- 某类事务出现纠偏 → 信任分回落 → 回到需请示状态

这不需要在第一版实现，但框架应该为这种演进留出空间。

## 6. 培训反馈环

整个培养过程的核心是让反馈环转起来。这是框架最优先要实现的事情——不是先把 Agent 做"聪明"，而是先让培训过程 work。

```
观察并提议 (observe_and_propose)
    ↓
老板反馈 (boss_feedback)
    ↓
提炼入册 (learn_to_handbook)
    ↓
执行并验证 (apply_and_verify)
    ↓
（循环）
```

### 四个环节

**观察并提议**：Agent 感知到外部事件，在工作台向老板提出处理方案——不直接执行。这是见习期的核心行为，也是成长期对新类型事务的默认行为。

**老板反馈**：老板用自然语言给出"行"或"不行"，以及修改意见。这是最自然的培训方式。

**提炼入册**：框架捕获老板的反馈，不仅修改当次行动，还要归纳出规则写入员工手册，并请老板确认。

**执行并验证**：下一次遇到同类事务时，Agent 按手册中的规则执行，并通过汇报让老板验证——手册里的规则是否真的符合老板的期望。

### 优先级

对于框架开发者来说，搭建顺序是：

1. 先让 observe_and_propose 跑通——Agent 能把外部事件翻译成提议
2. 再让 boss_feedback 跑通——老板的反馈能被正确捕获
3. 然后让 learn_to_handbook 跑通——反馈能被提炼为持久化的规则
4. 最后让 apply_and_verify 跑通——规则能在下次执行时被应用和验证

## 7. 工作台：一切的基础设施

上述所有机制——自然语言培训、员工手册、信任等级、培训反馈环——都建立在**工作台**这个基础设施之上。

工作台是用户与飞书原生 App Bot 的私聊，它承载了所有培养过程的交互：

- **草稿提交与确认**：Agent 在这里提议，老板在这里拍板
- **手册更新通知**：Agent 在这里呈现手册变更，老板在这里确认
- **复盘汇报**：Agent 在这里回顾今天做了什么，老板在这里点评
- **主动指令**：老板在这里随时下达新规则
- **冲突提醒**：Agent 在这里指出矛盾，老板在这里裁决

工作台的详细设计见 [工作台与事务设计](./2026-02-25-workbench-and-tasks.md)。

## 8. 框架的核心价值

之前的 Agent 框架（如 LangChain）关注的是如何连接工具。我们的框架关注的是**如何建立信任**。

很多 Agent 开发出来后，老板不敢让它进群——怕它乱说话。我们通过框架建立一套"从见习到转正"的培养路径，让老板逐步放心地交出代理权。

框架的核心不是让 Agent 更聪明，而是让培养过程更顺畅。一个培养机制好的"笨" Agent，比一个没有培养机制的"聪明" Agent 更有价值——因为前者会越来越好，后者永远让人不放心。
