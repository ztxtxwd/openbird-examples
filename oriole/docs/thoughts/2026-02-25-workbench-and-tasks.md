# 工作台与"事"的设计

> 上游文档：[通用 Agent 架构设计](./2026-02-25-general-agent-architecture.md)

## 工作台

工作台是用户与飞书原生 App Bot 的私聊，是老板和 Agent 的专属工作空间。

### 已确定

- 老板和 Agent 地位对等，双方都可以发起"事"、推进"事"、请求对方协助
- 工作台同时是 Agent 的配置界面——通过自然语言对话完成配置变更
- 工作台本身就是短期记忆——飞书即持久化层
- 定时任务队列也在工作台中（通过飞书定时消息实现）
- Agent 通过飞书 Open API（@larksuite/node-sdk）以 Bot 身份在工作台中发言、回复、编辑消息
- **消息编辑原则**——Agent 可以编辑自己发出的消息（如更新"事"的状态摘要），但如果老板已经回复了某条消息，该消息不应再被编辑，因为老板的回复是对发出时刻内容的回应

### 待讨论

- 是否支持多个工作台（多个老板各一个？一个老板多个？）
- 工作台里的消息格式规范（纯文本？富文本卡片？）
- 工作台元信息的存储方式（本地 settings.json？Bot 自身状态？）

## "事"

协作的最小单位。不是"消息"，不是"会话"，而是一件需要被处理和闭环的事务。

### 已确定

- 一条消息可能包含多件事，一件事可能横跨多个会话
- "事"的识别和拆解由 Agent 完成
- Agent 的核心职责标准：事事有回应
- 信号与行动解耦——信号从会话 A 进来，Agent 可能在会话 B 行动；也可能什么都不做，只更新记忆
- **表示形式：飞书话题（thread）**——每件"事"对应工作台私聊里的一个话题。新建一件事 = 发一条消息 + 创建话题；追加信息 = 回复已有话题；老板和 Agent 都在话题内推进和沟通

### 待讨论


- "事"的状态模型（进行中 / 等待老板 / 已闭环 / ...）
- 状态如何标记和流转（消息内容？reaction？卡片？）
- "事"的生命周期——什么时候算闭环？谁来判断？
- 历史"事"的归档和清理
