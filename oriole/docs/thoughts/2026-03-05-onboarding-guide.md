# 新手引导设计

> 上游文档：[工作台初始化设计](./2026-03-05-workbench-initialization.md)、[Agent 产品哲学与培养范式](./2026-03-03-agent-philosophy-and-training-paradigm.md)

## 背景

工作台创建完成后，老板面对的是一个新的 Bot 私聊和一个陌生的 Agent。如果没有引导，老板不知道能做什么、该说什么。

新手引导的目标：让老板在几分钟内走通关键流程，建立对 Agent 的初步认知和信任。

## 设计原则

- **引导即培养**：引导过程本身就是第一次培养。通过引导，老板自然地进入"对话即培训"的范式，不需要额外学习"如何培训 Agent"
- **在飞书内完成**：所有引导都在工作台私聊中通过消息完成，不跳转外部页面，不弹出教程窗口
- **可跳过**：老板随时可以说"跳过"或直接开始使用，引导不阻塞正常功能
- **渐进式**：不一次性灌输所有功能，而是按使用频率和重要性分步引导

## 引导流程

### 第一步：自我介绍

工作台创建后，Agent 发送第一条消息：

```
你好老板，我是你的 Oriole 助手。

这个对话是我们的工作台——你在飞书上的事务，我会帮你处理。你有什么指示直接跟我说就行。

接下来我带你走一遍主要功能，大概两分钟。随时可以说"跳过"。
```

设计要点：
- 一句话说清楚工作台是什么
- 明确老板只需要在这里说话就行（零门槛）
- 给出时间预期
- 提供跳过出口

### 第二步：体验"观察并提议"

引导老板体验培训反馈环中最核心的环节——Agent 观察事件、提出方案、老板拍板。

```
我平时会帮你盯着飞书上的消息。有人找你的时候，我会先看看情况，然后在这里跟你商量怎么回。

比如——假设有人刚才问你："明天的会几点开始？"

我会这样跟你说：
"老板，张三在问明天的会议时间。我查了你的日历，明天下午 2 点有产品评审会。我准备回他：'明天下午 2 点，产品评审会。' 可以吗？"

你觉得行就说"行"，想改就直接告诉我怎么改。
```

设计要点：
- 用具体例子演示"观察并提议"的模式
- 让老板理解 Agent 不会擅自行动（见习期）
- 暗示老板的反馈方式：批准或纠偏

### 第三步：体验下达指令

引导老板尝试给 Agent 一个指令，体验"自然语言即配置"。

```
你也可以随时给我立规矩。比如：

- "李总的消息第一时间告诉我"
- "报销的事先去问财务小王"
- "回消息别太正式，口语化一点"

试试看？给我一条你的工作习惯或者偏好。
```

这一步是互动式的——等待老板真正输入一条指令。Agent 收到后：

1. 将指令提炼为员工手册条目
2. 在工作台确认

```
收到。我记下了：「{老板的指令}」

这条我写进了员工手册，以后会按这个来。如果之后想改，随时跟我说就行。
```

设计要点：
- 老板第一次感受到"说一句话就能配置 Agent"
- 员工手册从这里开始生长
- 展示了手册更新的透明化确认机制

### 第四步：收尾

```
好了，基本就是这样：

1. 有人找你 → 我会在这里跟你商量
2. 你有指示 → 直接在这里说
3. 我越来越了解你 → 处理得越来越顺手

现在开始正式上岗。有什么事随时招呼我。
```

设计要点：
- 三句话总结核心交互模式
- 正式开启见习期

## 引导状态管理

- 引导进度记录在本地存储中，标记当前进行到第几步
- 老板说"跳过"时，标记引导完成，不再触发
- 老板长时间不响应时（如离开飞书），Agent 不催促，等老板回来后从上次中断的步骤继续
- 引导完成后不会再次触发

## 引导与培训反馈环的关系

引导流程本身就是培训反馈环的第一轮：

| 引导步骤 | 对应的反馈环环节 |
|----------|-----------------|
| 第二步：体验观察并提议 | observe_and_propose → boss_feedback |
| 第三步：体验下达指令 | boss_feedback → learn_to_handbook |
| 收尾后进入正常工作 | apply_and_verify（循环开始） |

引导结束后，老板已经自然地进入了培养模式，不需要额外过渡。

## 待讨论

- 引导消息的具体措辞需要反复打磨（当前为初稿）
- 是否需要在引导中收集老板的基础信息（如姓名、称呼偏好）
- 引导中的模拟场景是否使用真实的飞书数据（如真的去查日历），还是纯演示
- 二次引导——老板使用一段时间后，是否需要介绍进阶功能（如定时任务）
- 引导中如果老板给出的第一条指令比较复杂或含糊，Agent 如何处理
