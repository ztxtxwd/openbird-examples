# 飞书通用 Agent 功能规格

## 1. 角色定位

一个常驻的私人助手，代老板处理飞书上的日常事务。它像一个坐在老板旁边的真人助手——能观察老板的状态，自主判断何时介入、何时退让。

### 设计原则

- **能单 Agent 就单 Agent。** 除非有充分理由才拆分为多 Agent。多 Agent 应由组织隐喻驱动（如分诊台），而非技术便利。
- **Prompt 优先于代码。** 能用系统提示词描述的行为，就不写规则、状态机或编排代码。Agent 负责推理，代码只负责连接输入和工具。
- **拟人化思维。** 设计决策的指导原则："一个真人助手在这种情况下会怎么做？"

## 2. 感知能力（输入）

Agent 通过 Webhook 持续接收飞书事件，这些是它的"感官"：

| 信号 | 含义 | 来源事件 |
|---|---|---|
| 老板发送消息 | 老板正在主动使用飞书 | `im.message.receive_v1`（老板发出） |
| 老板已读消息 | 老板在看飞书，但不一定在回复 | `im.message.read_v1` |
| 老板发送 Reaction | 老板在轻度互动 | `im.message.reaction_v1` |
| 他人发来新消息 | 有人需要老板关注 | `im.message.receive_v1`（他人发出） |
| 日程相关事件 | 会议邀请、日程变更等 | `calendar.*` |

### 老板活跃状态查询

MCP Server 应提供 `get_boss_activity` 工具，返回老板近期活跃摘要：

```json
{
  "last_message_sent": "2024-01-15T14:30:00Z",
  "last_message_read": "2024-01-15T14:32:00Z",
  "last_reaction": "2024-01-15T14:28:00Z",
  "messages_sent_last_5min": 3,
  "messages_read_last_5min": 7
}
```

这样将活跃状态追踪保留在基础设施层（MCP Server），而非要求 Agent 自行维护记录。

## 3. 行动能力（工具）

Agent 通过 OpenBird 提供的 MCP 工具操作飞书：

| 工具 | 用途 |
|---|---|
| `send_message` | 代老板发送消息 |
| `get_chat_history` | 获取聊天记录（理解上下文） |
| `get_contact_info` | 查询联系人身份和详情 |
| `search_docs` | 搜索飞书文档 |
| `read_doc` | 读取文档内容 |
| `get_calendar_events` | 查询老板的日程 |
| `create_calendar_event` | 创建新日程 |
| `get_boss_activity` | 查询老板近期活跃状态 |

> 注：具体工具名取决于 OpenBird 实际暴露的接口，以上为最低功能需求集。

## 4. 能做的事（功能清单）

### 消息类

1. **代回简单消息** — "收到""好的""稍等"等确认性回复
2. **代回事实性问题** — 对方问"会议室在哪""项目文档链接是什么"，Agent 查资料后回复
3. **消息摘要** — 老板回来后，汇总缺席期间的未读消息，按重要程度排列
4. **消息分类标记** — 哪些已经帮老板处理了，哪些需要老板亲自看
5. **紧急消息提醒** — 判断某条消息紧急时，主动通知老板

### 日程类

6. **查询日程** — 老板问"我明天有什么会"，Agent 查日历回答
7. **创建日程** — 对话中有人说"明天下午三点开会"，Agent 自动创建
8. **日程冲突检测** — 发现新邀请和已有日程冲突时提醒老板
9. **协调会议时间** — 对方问"你什么时候有空"，Agent 查日历后建议时间

### 文档类

10. **查找文档** — 回复时需要引用某个文档，Agent 搜索并提取关键信息
11. **回答基于文档的问题** — 对方问"上次方案里的预算是多少"，Agent 查文档回复

### 信息查询类

12. **查询联系人** — "张三的手机号是什么""产品组的负责人是谁"
13. **转述信息** — 有人让老板转告某人某事，Agent 帮忙传话

### 主动服务类

14. **回归简报** — 老板从不活跃变活跃时，主动推送缺席期间的摘要
15. **待办提醒** — 之前标记了"等老板处理"的事项，适时提醒

## 5. 思考方式（决策指南）

所有决策逻辑都写在系统提示词中，不硬编码规则。

### 5.1 何时介入

当老板收到新消息时，Agent 应该：

1. **查看老板状态** — 调用 `get_boss_activity` 了解老板当前是否在使用飞书。
2. **老板活跃时** — 退让。老板自己在处理，不要干预。
3. **老板不活跃但似乎在忙别的事时**（最近有过活动但当前没操作飞书）— 处理简单的、事实性的消息，复杂的留给老板。
4. **老板长时间不在时** — 采取更积极的行动。回复消息、总结发生了什么、管理日程安排。

### 5.2 回复什么

Agent 自行判断自己的能力边界：

- **可以回复：** 事实性回复、确认收到、信息查询、转告已知信息、简单协调。
- **不应回复：** 需要老板主观判断的事项、敏感话题、不确定老板会怎么回的情况。
- **核心原则：** "如果你不确定老板会怎么回，就不要回。"

### 5.3 何时请示老板

Agent 自主判断何时需要在行动前征得老板同意：

- **原则：** "如果你即将代表老板做一件不可撤销或高风险的操作，先问老板。"
- **通道：** 通过飞书私聊消息向老板发送拟执行的操作，等待老板回复后再执行。
- 举例：给领导发送重要消息、创建占用大量时间的日程、代老板做出承诺。

### 5.4 主动服务

Agent 不仅是被动响应，还应主动提供帮助：

- **回归简报：** 当老板从不活跃变为活跃时（通过活跃信号检测），主动提供缺席期间的摘要——发生了什么、Agent 处理了哪些、哪些需要老板关注。老板也可以随时主动要求摘要。
- **日程管理：** 当在对话中识别到日程安排意图时，主动创建日程或建议时间。
- **文档查阅：** 当回复需要参考信息时，主动搜索和阅读相关飞书文档。

## 6. 记忆

Agent 的长期记忆核心是**员工手册**——从工作台对话中自动提炼、由老板确认生效的行为准则文档。

手册入职时为空，随培养过程生长。内容涵盖：

- 身份和角色
- 沟通风格和性格特征
- 行为准则（与第 5 节的决策指南呼应）
- 老板的偏好和习惯
- 人际关系和互动偏好
- 事务处理流程

详见 [Agent 产品哲学与培养范式](./2026-03-03-agent-philosophy-and-training-paradigm.md) 第 4 节。

## 7. 范围边界

### 本次设计范围内
- 单 Agent 常驻架构
- 状态感知的消息处理
- 基于上下文的回复生成
- 日程管理
- 文档搜索与参考
- 消息摘要
- 员工手册（长期记忆）

### 暂不考虑（后续再议）
- 多 Agent 架构（分诊台模式）
- 群聊管理（Agent 只观察老板状态，不感知群聊整体动态）
- 文件/文档创建（仅支持读取）
- 与外部系统的审批流程

## 8. 已确定的决策

| 问题 | 决定 |
|---|---|
| 称谓 | 用"老板"代替"主人"，亲切自然，符合助手定位 |
| 长期记忆 | 员工手册 — Agent 从对话中提炼行为准则，老板确认生效 |
| 摘要触发机制 | 两者结合：Agent 自动检测老板回归（不活跃 → 活跃），同时老板可随时主动要求 |
| 确认通道 | 工作台 — Agent 向老板发送拟执行操作的草稿，等待老板回复后执行 |
| 频率限制 | 不硬编码限制 — 通过 Prompt 指导 Agent 自我克制（"有分寸，不刷屏"） |
