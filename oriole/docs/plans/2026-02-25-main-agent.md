# 主 Agent 设计

> 上游文档：[通用 Agent 架构设计](./2026-02-25-general-agent-architecture.md)、[飞书通用 Agent 功能规格](./2026-02-21-feishu-agent-design.md)

## 定位

处理管线的第三层。被工作台变化触发，看全局状态，决定行动。使用强 LLM。

## 已确定

### 触发条件

工作台有新变化时触发（包括分类 Agent 追加信息、老板在工作台发消息、定时消息发出等）。

### 行动空间

- 需要查资料 → 用"眼"（搜索文档、读代码、查数据库、调 API）
- 需要执行 → 用"手"（发消息、建日程、调 API）
- 需要跟老板沟通 → 在工作台里发消息
- 需要定时 → 编辑定时消息队列
- 不需要行动 → 结束本轮

### 决策指南（来自功能规格）

**何时介入：**
- 老板活跃时 → 退让
- 老板不活跃但似乎在忙别的事 → 处理简单的、事实性的消息
- 老板长时间不在 → 采取更积极的行动

**回复什么：**
- 可以回复：事实性回复、确认收到、信息查询、转告已知信息、简单协调
- 不应回复：需要老板主观判断的事项、敏感话题、不确定老板会怎么回的情况
- 核心原则："如果你不确定老板会怎么回，就不要回。"

**何时请示老板：**
- 原则："如果你即将代表老板做一件不可撤销或高风险的操作，先问老板。"
- 通道：在工作台里向老板发送拟执行的操作，等待老板回复后再执行

### Barge-in 机制

Agent 执行动作过程中如果工作台有新变化：
- 已执行的动作不撤回
- 未执行的动作丢弃
- 基于最新状态重新思考和行动

### 自触发处理

Agent 更新工作台也会触发新一轮处理：
- Agent 醒来，看到最新消息是自己发的
- 判断：正在等老板回复 / 没有新的事需要处理
- 结束本轮

不需要特殊的防循环机制。

## 待讨论

- 每次触发时 system prompt 的结构和拼装方式
- 上下文窗口的内容选择策略（放哪些"事"？放多少历史？）
- context window 溢出时的处理
- Barge-in 的技术实现（如何检测新变化？工具调用的原子性？）
- 并发处理——多个"事"同时需要行动时的优先级和调度
- 错误处理——LLM 调用失败、工具调用失败时的恢复策略
- "老板活跃状态"在新架构中的获取方式（功能规格里提到 `get_boss_activity`，在工作台模式下如何适配）
- 主动服务的触发（回归简报、待办提醒）在工作台模式下如何实现
