# 分类 Agent 设计

> 上游文档：[通用 Agent 架构设计](./2026-02-25-general-agent-architecture.md)

## 定位

处理管线的第二层。位于规则过滤之后、主 Agent 之前。使用轻量 LLM。

只做一件事——判断这个刺激跟什么"事"有关。

## 已确定

### 输入

经过第一层规则过滤后的外部刺激（飞书事件）。

### 输出（三选一）

1. 属于工作台里已有的事 → 回复对应话题，把信息追加到工作台
2. 是一件新的事 → 在工作台里发消息并创建新话题
3. 不构成任何"事" → 丢弃（第二层过滤）

### 工具使用

分类 Agent 需要调用工具来补充上下文，因为刺激本身携带的信息往往不足以做出判断。

典型场景：张三在某个群里只发了"好的"两个字——单凭这条消息无法判断它属于哪件事，必须去查这条消息所在会话的上下文。

**读取工具**（用于理解刺激的上下文）：主 Agent 能用的只读工具都可以用（如 `get_chat_history`、`get_contact_info`、`search_docs`、`read_doc` 等）。不限制调用次数——分类准确性优先于调用成本。

**工作台写入工具**（用于将分类结果落到工作台）：
- `send_message` — 在工作台群里发消息（如新建一件"事"的首条消息）
- `create_thread` — 给工作台群里的某条消息创建话题
- `reply_thread` — 回复话题
- `edit_message` — 编辑已有消息

### "事"的表示形式

工作台里的每件"事"对应飞书群里的一个**话题（thread）**。新建一件事 = 发一条消息 + 创建话题。

### 追加信息的策略

当有新信息需要归入已有的"事"时：
1. **优先编辑原消息** — 更新话题首条消息的内容，使其始终反映这件"事"的最新全貌
2. **无法编辑时回复话题** — 作为兜底，在话题内追加回复

### 上下文窗口

除了刺激本身和工具调用结果外，上下文窗口中应包含工作台里最近 N 件"事"的摘要，使分类 Agent 知道当前有哪些事在进行中，才能判断新刺激是否属于其中之一。

### 约束

- 使用轻量 LLM，按职能分脑（分类用轻量模型，决策用强模型）
- 不做决策，不代老板在外部会话中执行动作，只做分类和路由
- 写入范围仅限工作台——用于组织"事"，不在外部会话中发消息、不建日程等
